## Why the card grows in height on hover

This rule is intended to create a dark translucent overlay on top of the article link. It uses `position: absolute` with `width: 100%` and `height: 100%`, which requires the **parent** (`.masonry-article-link`) to be a containing block (i.e. have `position: relative` or similar).

Look at the `.masonry-article-link` rule:

```css
.masonry-article-item .masonry-article-link {
  position: relative;
  display: block;
  margin: 0 auto;
  border-bottom: none;
}
```

Good — it is `position: relative`. So the overlay *should* be absolutely positioned inside the link. **But** this rule only applies to `.masonry-article-link` that sits inside `.masonry-article-item`.

Your markup is different:

```html

</div>
```

The card is a `.card.bottom` wrapping a `.lg-item` anchor (per the selector `.card.bottom > a.lg-item` in the lightGallery CSS), **not** a `.masonry-article-item > a.masonry-article-link`. So:

1. The `.masonry-article-hover` rule (which would lay the overlay on top) never matches your card's anchor.
2. If your card contains a `<div class="masonry-article-hover">` (or equivalent), it inherits only the top part of the rule — `display: flex; position: absolute; width: 100%; height: 100%` — but because the ancestor anchor is NOT `position: relative` in your layout, the overlay is treated as flow content / positions relative to a different ancestor, effectively pushing the card to grow.

Even simpler explanation that matches what you're seeing: the overlay element `.masonry-article-hover` has `display: flex` and `height: 100%` applied even when `opacity: 0`. For that element to sit *on top of* the card (as an overlay) instead of *inside the flow* (which would add its height to the card), **its closest positioned ancestor must be the card itself**. In your `.card.bottom` markup that positioned ancestor is missing, so the overlay is being laid out in normal flow and its `height: 100%` plus flex sizing enlarges the card on hover — exactly the symptom you described.

A second contributor: the rules `.masonry-article-item .masonry-article-link:hover .masonry-article-hover { opacity: 1 }` trigger a layout reflow on hover because the overlay (or a sibling) may switch from `display: none`-ish behavior to `display: flex` via some cascade in your local overrides. Since the overlay is in the flow (not absolute relative to the card), growing opacity coincides with its flex box taking its natural height on hover.

**Why the card enlarges on hover**

The height growth is a CSS issue, not a masonry.js issue. The masonry library and `_caption.scss` are not involved — they only compute grid positions and style `<caption>` elements respectively. The culprit is in the masonry `uno.css`:

1. The existing `.masonry-article-hover` overlay relies on `width: 100%; height: 100%` but has no `top/right/bottom/left: 0`. 2. `.card.bottom` itself has no `position: relative`, so it isn't a containing block — any absolute child escapes upward.
3. The inner `a.lg-item` has default browser hover behavior (text-decoration / border oscillation) that adds 1–2 px of height.
