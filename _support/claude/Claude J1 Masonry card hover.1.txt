The cause is the CSS `background` **shorthand** in your hover rule.

Your original rules are:

```css
.lg-uno-video-icon .big-play-button {
  position: absolute; top:0; left:0; bottom:0; right:0;
  background: url(icons/play-button.svg) no-repeat center center;
}
.lg-uno-video-icon:hover .big-play-button {
  background: url(icons/play-button-hover.svg) no-repeat center center;
}
```

When a CSS **shorthand** property appears in a rule, every sub-property it governs that isn't explicitly listed is **reset to its initial value** for that rule. `background` is a shorthand for eight sub-properties (`background-image`, `background-position`, `background-size`, `background-repeat`, `background-origin`, `background-clip`, `background-attachment`, `background-color`).

The hover declaration omits `background-size`, `background-origin` and `background-clip`, so on hover those three silently snap back to their initial values (`auto`, `padding-box`, `border-box`). If *anything* upstream — a user-agent rule, a lightGallery rule, or a parent rule — had tweaked those sub-properties (and lightGallery's own styles do touch `background-*` on several inner elements), the reset causes the hover SVG to render at a different effective box or origin than the non-hover SVG, and it visually jumps off-center.

A secondary contributor: `.lg-uno-video-icon` has no `position` declared in the original file. An absolutely-positioned child with `top/left/bottom/right: 0` will only fill *that* parent if the parent is a positioning context. Otherwise it climbs to the nearest positioned ancestor, and any hover-triggered layout change on an intermediate element can shift it.

## The fix (all changes marked `claude - masonry card hover #1`)

1. **`.lg-uno-video-icon { position: relative; }`** — guarantees the button is positioned relative to the icon wrapper, not some farther ancestor.
2. **Replaced the `background` shorthand with longhand properties** (`background-image`, `background-repeat`, `background-position`, `background-size`) in both the base and `:hover` rule. The hover rule now changes only the image URL; nothing else resets.
3. **Restated `top/left/bottom/right: 0`** in the `:hover` rule as a defensive belt-and-braces measure.

The `masonry.js` adapter itself is unrelated to this — the issue was purely in the CSS theme file.
