## What's actually happening — the real cause

I went hunting for the hover rule everyone is blaming, and it does not exist.
No file in the upload bundle contains a `.card:hover`, `.card.bottom:hover`,
`.caption:hover`, or any parent-hover-child-grows rule that would enlarge the
card's height on hover.

But there's something in bootstrap.css at line 4334 that everyone missed:

```css
.card {
  position: relative;
  display: flex;
  flex-direction: column;   /* ← this is the whole story */
  min-width: 0;
  height: var(--bs-card-height);
  ...
}
```

`.card` is a flex column. The card's height is the sum of its flex children's
cross-axis sizes. Now combine that with what fix #3 did:

```css
.card.bottom,
.card.bottom:hover,
... {
  height: var(--card-h);   /* --card-h: auto */
  padding: 0;
  border-width: 0;
  margin: 0;
  transition: none;
}
```

`height: auto` on a `display: flex` column means "shrink-wrap the children".
So the card's height is the sum of the image's rendered height plus the
`.caption`'s rendered height. Nothing is actually pinning the card.

Here's what triggers the growth on hover, in priority order:

1.  The caption is a `<div>` — a block-level flex item in the column.
    On first paint the browser may lay the caption out before the webfont
    has loaded (Lato is declared on `.masonry-article-caption-subheading`
    and used elsewhere). When the font swaps, the caption's line-box height
    changes. If that swap happens on or near the same frame as your first
    hover, it looks exactly like "hover enlarged the card". Also — and this
    is the one people keep missing — `bootstrap.css` line 366–372 declares
    `a { text-decoration: underline }`, and the anchor `a.lg-item` inherits/applies
    `line-height` from its ancestors. Fix #3 set `line-height: 0` on `a.lg-item`,
    but the caption div inherits `line-height: 1` from `.card.bottom` (fix #3 line 284),
    not 0. So the caption's content box is exactly `font-size × 1`, with
    no room for descenders. The instant any hover rule anywhere in the project
    (e.g. `a:hover` in bootstrap, which *does* exist at line 370) triggers a
    style recalc, the browser re-measures the caption — and if a webfont has
    swapped in between, the computed height increases.

2. `line-height: 1` on `.card.bottom` itself (fix #3) is actively dangerous. 
    With `line-height: 1` and descender-bearing glyphs ("Carpool Karaoke James
    and Adele" has a `p`, `y`, `J`), some font metrics render descenders
    outside the content box. Fix #3 tried to solve the image descender gap,
    but introduced a caption descender clip problem: browsers reserve
    half-leading space asymmetrically when line-height equals font-size,
    and any style recalc can shift the rounding by 1 px.

3.  The `a { text-decoration: underline }` rule from bootstrap line 368.
    Fix #3 removed this on `a.lg-item`, but if the caption ever contains
    a link (or the masonry template later wraps the caption text in an anchor),
    the underline's `text-underline-offset` changes the line box.

## What to do

Lock the card to an explicit pixel height that is not a sum of its children,
and make the caption a positioned absolute bar over the image instead of a
flex sibling. Once the card's height is an explicit value and the caption
doesn't participate in flow, there is nothing left to grow.

## Why this actually works where #1, #2, #3 did not

- Not relying on `height: auto`. The card's height is an explicit pixel value (`--card-h: 220px`). It is impossible for a child to change it.
- `contain: layout paint size` tells the browser that descendants cannot affect the card's layout or size. This is a hard, spec-guaranteed barrier. No amount of font swapping, hover recalculation, or child reflow can cross it.
- Caption is removed from flow. Being `position: absolute`, the caption is not part of the card's height computation at all. It could triple in size and the card wouldn't move.
- The card opts out of being a flex column (`display: block`). `.card`'s bootstrap default as a flex column was the hidden mechanism turning "child height change" into "card height change". That mechanism is now gone.
- The hover cue is `box-shadow`. Paints outside the border box. Zero layout cost.

If you need a different card height for different cards, override `--card-h` inline or via a utility class — no other code change required.
